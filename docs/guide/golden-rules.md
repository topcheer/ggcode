# Karpathy 4 条黄金定律（通用版）

## 1. 想清楚再做 — 先定义问题，再定义方案

不要在方案里藏模糊地带，不要替别人做未经确认的假设。开始动手前：

- 把底层假设说清楚；如果不确定，直接标出来让人确认。
- 如果有多种解读方式（比如"优化"是指提升效率还是降低成本），全部列出来，不要默认别人按你的理解去执行。
- 如果发现更简单的做法能解决同一个问题，先说出来，而不是堆复杂度。
- 有歧义就停下来问，不要在产出物里写"待定"然后指望别人猜。

## 2. 简单优先 — 最小范围解决真实问题

每次只做一件事，不做推测性的扩展：

- 不加未验证的东西；为"以后可能用"而提前准备时，必须有事实或数据支撑，而不是凭空假设需求会膨胀。
- 不为了显得周全而塞进大量边界情况，真正高概率的场景覆盖到就行。
- 如果一页能讲清，不要用十页；如果一步能完成，不要拆成三步。
- 自问："这件事的核心价值是什么？去掉哪些部分它就不成立了？"先把那些去掉的部分拿掉。

## 3. 外科手术式变更 — 只改该改的，别顺手重做

做改动时：

- 只改这次要解决的问题，不要借机重排整个结构或调整无关的部分。
- 保持和现有体系的风格一致，即使你觉得旧方式不够好，也不要顺手改掉。
- 如果自己的改动产生了连锁影响（比如改了一处导致其他地方要跟着变），只处理本次必须跟动的部分。
- 发现环境里已有的不合理之处，记录下来留待后续处理，不要在这一次顺手清理。

## 4. 目标驱动 — 验收标准先于方案

把目标翻译成可验证的成功条件，而不是只描述要做什么：

- **每件事必须绑定验收标准**，不能只有行动描述没有验证方式。验收标准写不出的事，说明这件事本身还没想清楚，先补清楚再动手。
- 验收标准应包含：
  - **触发场景**：在什么情境下需要这个结果
  - **正常用例**：前提条件满足时的预期结果
  - **异常用例**：输入有误、条件缺失、突发中断等边界情况的处理
- 如果一件事只写了正常用例、没写异常用例，它就不算完成。
- "提升业绩" → "关键指标提升 X%，用前后对比数据验证"
- "优化流程" → "完成时间从 X 降到 Y，用实测计时验证"
- "修复问题" → "复现路径是 ABC，处理后该路径不再触发"

多步骤的事情要写出简要计划，每步都标注怎么验证：

```
1. [事项] → 验证：[具体检查方式]
2. [事项] → 验证：[具体检查方式]
3. [事项] → 验证：[具体检查方式]
```

清晰的目标让各方独立对齐，模糊的目标只会带来反复沟通。
