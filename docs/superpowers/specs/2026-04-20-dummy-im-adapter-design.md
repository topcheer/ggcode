# Dummy IM Adapter for Automated Knight Evaluation

**Date:** 2026-04-20
**Status:** Draft
**Context:** `docs/research/knight-evaluation-plan.md` Phase 2-4 requires a programmatic "fake user" to drive multi-turn conversations with ggcode agent, including ask_user responses, without human involvement.

## Problem

Knight evaluation (smoke test, A/B benchmark, soak run) needs:

1. **Multi-turn interaction** вҖ” build -> test -> fix -> retest is not a single prompt
2. **ask_user response** вҖ” agent may ask clarifying questions mid-task; evaluation must answer
3. **LLM-generated prompts** вҖ” task descriptions are generated by LLM, not hand-written
4. **Unattended execution** вҖ” no human at keyboard

Current `ggcode -p` is single-turn only. TUI requires a human. Real IM adapters require external platform accounts.

## Decision

Add a `dummy` IM adapter that starts a localhost HTTP server, allowing an external orchestrator to:

- Send messages as a fake user (`POST /send`)
- Receive all agent output via SSE (`GET /events`)
- Respond to ask_user by sending another message (DaemonBridge already routes to pending ask channel)
- Collect structured metrics

All communication uses the existing `Sink`/`Bridge` interfaces. Agent and Knight cannot distinguish dummy adapter from a real IM platform.

## Components

### 1. DummyAdapter (`internal/im/dummy_adapter.go`)

Implements `Sink` + `startableSink`.

```go
type dummyAdapter struct {
    name    string
    manager *Manager

    server    *http.Server
    listener  net.Listener
    portFile  string           // path to write discovered port

    sseBroker *sseBroker       // fan-out SSE event distribution
    metrics   *EvalMetrics     // per-task metrics collection

    mu sync.Mutex
}

func (a *dummyAdapter) Name() string { return a.name }

func (a *dummyAdapter) Send(ctx context.Context, binding ChannelBinding, event OutboundEvent) error {
    // 1. Record metrics
    a.metrics.RecordEvent(event)
    // 2. Push to SSE broker
    a.sseBroker.Push(event)
    return nil
}

func (a *dummyAdapter) Start(ctx context.Context) {
    // 1. Auto-bind channel (skip pairing вҖ” no real IM user to verify)
    // 2. Start HTTP server on listener
    // 3. Write port to portFile (atomic: write tmp then rename)
    // 4. Block until ctx.Done(), then cleanup portFile
}
```

#### Channel Binding (Critical Path)

Dummy adapter must establish a `ChannelBinding` before any message can flow through `HandleInbound`. Unlike real IM adapters that go through pairing, dummy adapter auto-binds in `Start()`:

```go
func (a *dummyAdapter) autoBind() error {
    binding := ChannelBinding{
        Workspace:  workingDir,
        Platform:   PlatformDummy,
        Adapter:    a.name,
        TargetID:   "eval-user",
        ChannelID:  "eval-channel",
        BoundAt:    time.Now(),
    }
    return a.manager.BindChannel(binding)
}
```

This is called after `RegisterSink` but before the HTTP server starts accepting requests.

### 2. HTTP Server (`internal/im/dummy_server.go`)

Endpoints:

#### `POST /send` вҖ” Simulate user message

```
Request:  {"text": "run make test and fix failures"}
Response: {"status": "ok", "message_id": "msg_xxx"}

Query params:
  ?reset_metrics=true  вҖ” reset EvalMetrics for a new task
```

Handler constructs an `InboundMessage` and calls `mgr.HandleInbound()`. If `DaemonBridge` has a pending ask_user, the message fills the pending channel and unblocks the agent.

**Concurrency:** `/send` handler serializes requests via a mutex. Only one message is processed at a time. The orchestrator must NOT call `/send` concurrently. Response blocks until the message has been submitted to the bridge.

**Message ID:** Server generates a UUID for each message's `Envelope.MessageID`. Client may optionally include `client_message_id` in the request body for idempotent retries (server deduplicates within the 5-minute window).

**Shutdown token:** The server generates a random token at startup, written to the port file alongside the address. `POST /shutdown` requires `Authorization: Bearer <token>` header.

#### `GET /events` вҖ” SSE event stream

SSE format:
```
event: text
data: {"kind":"text","content":"..."}

event: tool_result
data: {"kind":"tool_result","tool":"bash","result":"...","is_error":false}

event: approval_request
data: {"kind":"approval_request","tool":"ask_user","title":"...","questions":[...]}

event: knight_report
data: {"kind":"knight_report","content":"..."}

event: round_done
data: {"kind":"round_done","tool_calls":5,"tool_successes":4,"tool_failures":1,"elapsed_ms":12340}

event: metrics
data: {"tool_calls":{"bash":3,"read":2},"ask_user_count":1,...}
```

SSE broker maintains a ring buffer (default 1024 entries). Each event carries a monotonically increasing `seq` field. New SSE connections receive:
1. `event: hello` with `{"last_seq": N}` so the client knows the current position
2. Replay of buffered events from oldest available seq

Client can detect gaps: if replay starts at seq=50 but last known was seq=40, events 40-49 were lost.

`approval_request` events are pinned in the buffer (never evicted by ring rotation) since they block agent execution.

Heartbeat: every 15 seconds, send SSE comment frame `: keepalive\n\n` to detect dead connections.

#### `GET /status` вҖ” Current state snapshot

```json
{
  "bindings": [...],
  "knight": {"enabled":true,"status":"running","budget_used":50000},
  "metrics": {"total_tool_calls":12,...}
}
```

#### `GET /healthz` вҖ” Readiness check

Returns `{"status":"ok"}`. Orchestrator should poll this after startup until it succeeds, rather than relying solely on the port file.

#### `POST /shutdown` вҖ” Graceful shutdown

### 3. EvalMetrics (`internal/im/dummy_metrics.go`)

```go
type EvalMetrics struct {
    SessionStart     time.Time
    SessionEnd       time.Time

    UserMessages     int
    AskUserCount     int
    AskUserLatencyMs map[string]int64   // question_id вҶ’ latency

    ToolCalls        map[string]int      // tool_name вҶ’ count
    TotalToolCalls   int
    ToolErrors       int
    ToolErrorsByTool map[string]int      // tool_name вҶ’ error count
    ReworkCount      int                 // consecutive same-tool calls detected
    Rounds           int

    InputTokens      int
    OutputTokens     int

    KnightReports    int
    StagedSkills     int

    ElapsedMs        int64
}
```

Collection happens in `Sink.Send()`:

| OutboundEventKind | Action |
|---|---|
| `tool_result` | ToolCalls[name]++, TotalToolCalls++, ToolErrorsByTool[name]++ if error |
| `approval_request` | AskUserCount++, record start time |
| `text` | Heuristic detection: if text starts with "рҹҢҷ " prefix вҶ’ KnightReports++; if text contains round summary markers вҶ’ Rounds++ |

#### SSE Event Type Detection

Dummy adapter uses heuristics to distinguish SSE event types, since `OutboundEvent.Kind` does not have dedicated knight_report or round_done values:

- **knight_report**: `OutboundEventText` where text has prefix `"рҹҢҷ "` (set by `IMEmitter.EmitKnightReport`)
- **round_done**: `OutboundEventText` where text matches the round summary pattern (set by `IMEmitter.EmitRoundSummary`: includes tool call counts)
- **approval_request**: `OutboundEventApprovalRequest` вҖ” already a dedicated kind

#### Token Usage Collection

Token counts are NOT available in `Sink.Send()`. Two options were considered:
1. Add `TokenUsage` field to `OutboundEvent` (would require modifying emitter + daemon_bridge)
2. Accept limitation and collect tokens separately

**Decision: Option 2 вҖ” deferred token collection.** The `/status` endpoint will read token usage from the agent's session store (JSONL files) after each task completes. This avoids modifying core components. If this proves insufficient, a future iteration can add `TokenUsage` to `OutboundEvent`.

EvalMetrics records `InputTokens` and `OutputTokens` as 0 during streaming, then fills them from session data when `GET /status?final=true` is called after task completion.

Output:
- Real-time: `metrics` SSE event after each `round_done`
- Final: JSON file at configured `metrics_path`
- CSV: one row per task, compatible with `metrics.csv` from `make knight-eval`

### 5. Configuration

```yaml
im:
  enabled: true
  adapters:
    dummy:
      enabled: true
      platform: dummy
      extra:
        listen_addr: "127.0.0.1:0"       # auto-assign port
        metrics_path: ".tmp/knight-eval/metrics.json"
        sse_buffer_size: "1024"
        port_file: ".tmp/dummy-adapter-port"  # includes shutdown token
```

Config values are read from `adapterCfg.Extra` map, consistent with existing adapters (QQ reads `appid`, Telegram reads `bot_token` from Extra). Helper functions `stringValue()`, `intValue()` extract with defaults.

Port file format: `<host>:<port>\n<shutdown_token>`. Written atomically (write tmp file, then rename). Cleaned up on process exit. For multi-instance A/B testing, each instance uses a different config with distinct `port_file` and `metrics_path`.

### 6. Eval Orchestrator (`scripts/eval/`)

```
scripts/eval/
в”ңв”Җв”Җ run_eval.py          # main orchestrator

в”ңв”Җв”Җ llm_client.py        # LLM call wrapper (prompt gen + ask_user answers)
в””в”Җв”Җ requirements.txt     # httpx, openai or anthropic
```

Flow per task:
1. LLM generates concrete prompt from task template
2. `POST /send?reset_metrics=true` with generated prompt
3. Consume SSE events:
   - `approval_request` вҶ’ LLM generates answer вҶ’ `POST /send`
   - `round_done` вҶ’ check if agent finished
   - `knight_report` вҶ’ record
4. `GET /status` вҶ’ collect final metrics
5. Append row to `metrics.csv`

Per evaluation plan requirements:

Knight Score formula (from evaluation plan):
```
Knight Score
= 35% task success rate delta
+ 20% elapsed time improvement
+ 15% tool call reduction
+ 10% turn reduction
+ 10% useful skill rate
+ 10% operator trust score
```
