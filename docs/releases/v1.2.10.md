# ggcode v1.2.10

## Highlights

- **Data race fixes**: Fixed two map concurrent read/write races that could cause panics under specific concurrent conditions — `analyzedSessions` in Knight session analysis and `team.Teammates` in Swarm teammate snapshots.
- **Harness snapshot atomicity**: Task and release plan snapshots now use atomic writes (write-to-temp + rename), preventing partial JSON files on crash.

## Bug fixes

- **Fix**: `knight/analyzer.go` — `analyzedSessions` map accessed from both the runLoop goroutine (hourly auto-analysis) and arbitrary goroutines (manual `PerformSkillAnalysis`). Three unprotected access points now hold `k.mu`. Detected by `go test -race`.
- **Fix**: `swarm/manager.go` — `TeammateSnapshot()` read `team.Teammates` while only holding `m.mu`, but `SpawnTeammate()` writes the same map under `team.mu`. Now acquires `team.mu.RLock` for the read, matching the write-side lock.
- **Fix**: `harness/events.go` — `writeTaskSnapshot` and `writeReleasePlanSnapshot` used `os.WriteFile` (non-atomic). A crash mid-write left truncated JSON. Both now use the existing `atomicWriteJSON` helper already used by other functions in the same module.

## Test coverage

| Area | New tests |

All new tests pass under `-race` with 3 repetitions.

## Internal

- `internal/harness/events.go`: `os.WriteFile` → `atomicWriteJSON` in `writeTaskSnapshot` and `writeReleasePlanSnapshot`
