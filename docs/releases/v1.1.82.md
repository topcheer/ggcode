# ggcode v1.1.82

## Highlights

**WeChat iLink adapter** — zero-config QR scan authentication with dedicated TUI panel. No bot_token or corp_id needed; scan the QR code and you're online.

**DingTalk adapter fixes** — stream connection, subscription types, message parsing, and outbound reply routing all corrected.

**Critical IM fix: session history sync no longer leaks to all channels** — `SyncSessionHistory` was using `Emit()` (broadcast to all bound channels) instead of `SendDirect()` (target only the newly paired channel). When a new IM channel completed pairing, the entire session conversation history (tool calls, results, assistant text) was sent to **every** bound IM adapter, not just the new one.

## New Features

### WeChat iLink adapter
- QR scan auth with `go-qrcode` text rendering in TUI panel
- Auto-bind channel after successful QR scan
- Long-poll `getupdates` with dedicated HTTP client (separate from normal requests) to prevent EOF errors
- 4-digit verification pairing flow (same as QQ adapter)
- Platform display name in adapter list (`/listim`)

### IM output mode filtering (TUI mode)
- TUI now respects `im.output_mode` setting (`verbose`/`quiet`/`summary`) for tool_result events, same as daemon bridge
- `summary` mode: only sends final agent reply to IM, no individual tool results
- `quiet` mode: sends tool errors and final reply only
- `verbose` mode (default): sends all tool results (backward compatible)

## Bug Fixes

### DingTalk adapter
- **WebSocket stream**: added missing `ticket` field in WS URL and `Origin` header for handshake
- **Subscription type**: changed from `REQUEST` to `CALLBACK`/`EVENT` to match DingTalk SDK expectations
- **Bot callback text**: fixed `Text` field from `string` to nested struct (`{content: "..."}`) to match SDK's `BotCallbackDataTextModel`
- **Outbound reply**: `ChannelID` now stores `senderStaffId` (user ID) instead of `conversationId`; `oToMessages/batchSend` API includes `userIds` field
- **Webhook fallback**: `Send` method tries `sessionWebhook` first (cached from callback), falls back to REST API

### IM session history sync
- **Root cause**: `SyncSessionHistory` used `Emit()` which broadcasts to all bound channels. Fixed to use `SendDirect()` targeting only the newly paired channel
- All six panel bind functions (QQ, Telegram, Discord, Slack, DingTalk, Feishu) updated to pass the specific binding

### TUI /restart from IM
- Fixed `scheduleRemoteRestart` to return custom `quitMsg{}` instead of directly calling `tea.Quit()` — allows bubbletea to handle quit → `program.Run()` returns → restart executes properly

### Debug logging
- `maxMessageLen` increased from 150 to 4096 bytes — all debug log messages were being truncated at 150 bytes
- Added detailed debug logs for DingTalk callback processing, pairing flow, and Send path
- Added caller stack trace in emitter `EmitEvent` for text events

### WeChat adapter
- Added missing `Platform: PlatformWechat` to `publishState()` — `/listim` now shows correct platform
- `pollLoop` uses dedicated `lpClient` (40s timeout) separate from `httpClient` (30s) to prevent long-poll EOF

### TUI QR code display
- QQ and WeChat panel: QR code section prepended to top of panel body to avoid clipping when window height is insufficient
