# ggcode v1.1.32

## Highlights

- Adds a first-class IM runtime with QQ as the first built-in remote surface, including persisted
  workspace bindings, pairing, remote prompt delivery, and mirrored agent output.
- Polishes the TUI around the new remote workflow: `/qq` management, remote `/provider` and `/model`
  handling, better ask_user mirroring, and several input / scrolling fixes that make day-to-day use
  smoother.

## Fixes and improvements

### IM runtime and QQ integration

- Adds a new `internal/im` runtime with normalized inbound / outbound events, persisted workspace
  bindings, adapter state tracking, approval state, and pairing state storage.
- Ships a built-in QQ adapter with share-link generation, inbound text / image / voice handling, QQ
  message delivery, and STT integration hooks for voice attachments.
- Persists IM bindings in `~/.ggcode/im-bindings.json` and pairing state in `~/.ggcode/im-pairing.json`
  so the active workspace can restore its remote binding and pairing blacklist state.
- Binds the active TUI session into the IM runtime at startup and auto-starts only the adapter bound to
  the current workspace instead of eagerly connecting every configured QQ bot.
- Enforces workspace-level adapter exclusivity so one QQ adapter cannot be silently claimed by multiple
  directories at the same time.

### QQ panel and pairing UX

- Adds a dedicated `/qq` panel for bot creation, binding status, runtime health, channel clearing, and
  share-link / QR-based pairing.
- Moves QQ pairing back into the `/qq` panel instead of a separate fullscreen overlay.
- Switches terminal QR rendering to a compact `qrcode-terminal`-style layout so the code fits normal
  terminal heights more reliably.
- Localizes the QQ panel and related runtime / error copy for `en` and `zh-CN`.
- Shows IM adapter state in the sidebar so configured and active QQ bindings are visible without opening
  the panel.

### Remote command and ask_user workflow

- Lets remote IM users execute `/provider` and `/model` commands against the live TUI session and get
  an immediate summary or switch result back through the IM channel.
- Adds remote questionnaire answering for `ask_user`, including numeric choice parsing, multi-select
  parsing, freeform answers, and automatic progression to the next unanswered question.
- Mirrors `ask_user` prompts and other remote-safe status messages more cleanly instead of dumping raw
  tool chatter into the IM channel.

### IM delivery behavior

- Aggregates QQ / IM delivery per provider round so streaming chunks and tool-only rounds do not spam
  the remote channel.
- Mirrors the local user prompt to IM once, then emits a single final assistant text for the round.
- Preserves status-then-text ordering in the IM emitter and suppresses noisy sub-agent / tool-status
  spam that is not useful on the remote side.

### QQ outbound reply handling

- Removes local reply-window count / TTL gating from the QQ adapter and leaves duplicate / expiry
  enforcement to the QQ platform.
- Persists QQ `msg_seq` as `PassiveReplyCount + 1` for the current `msg_id`, so successive replies use
  a stable sequence instead of a timestamp-derived value.
- Keeps reply metadata intact on QQ `11255` failures instead of mutating local state during the failed
  send path.

### General TUI fixes

- Restores slash-command autocomplete behavior so selecting a slash command and pressing Enter executes
  it immediately instead of only echoing the command into the input box.
- Filters mouse wheel / cursor-report control sequences out of the composer so scrolling no longer
  injects garbage characters like `[` into the input field.
- Tightens viewport and preview scrolling behavior and keeps the loading spinner / shell-mode transitions
  consistent during command execution.
- Improves resumed session rendering with cleaner spacing and more compact command-tool history entries.

## Upgrade notes

- QQ auto-connect is now intentionally limited to the adapter bound to the current workspace.
- QQ remote continuation still depends on QQ passive-reply semantics. After a pause or restart, a fresh
  inbound QQ message is still the reliable way to establish a new reply context for that channel.
