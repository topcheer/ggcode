# v1.3.102

## Critical fixes

### Session index cross-process race condition

The session `index.json` was silently losing entries when multiple ggcode instances ran in the same workspace. The `sync.Mutex` only protected in-process concurrency — multiple OS processes sharing the same index file caused a classic TOCTOU race:

```
Process A: loadIndex() → [19 entries] → append A → saveIndex() → [20 entries]
Process B: loadIndex() → [19 entries] → append B → saveIndex() → [20 entries, A LOST]
```

Worse: concurrent writes could truncate the JSON, causing `loadIndex` to return empty, which made the next `updateIndex` overwrite all 307 entries with just 1.

**Fix (3 layers):**
- **Cross-process file lock**: New `lockIndexFile()` using `flock(LOCK_EX)` (Unix) / `LockFileEx` (Windows) wraps all read-modify-write cycles in `updateIndex`, `removeFromIndex`, and `repairIndex`.
- **Instant corruption recovery**: `loadIndex` now immediately rebuilds from disk when it detects corruption, instead of silently returning empty.
- **Disk fallback in workspace queries**: `ListForWorkspace` and `LatestForWorkspace` now trigger `repairIndex` (full disk scan) when the workspace isn't found in the index, preventing false "no sessions" results.

### Microcompact removed — simpler and more reliable compaction

The `Microcompact()` feature (in-place truncation of old `tool_result` blocks) was causing subtle issues: it modified live messages during background precompact, causing snapshot mismatches that rejected valid compaction results. It has been **removed entirely**.

- Precompact now calls `Summarize()` directly (no `CheckAndSummarize` gate that could skip due to token estimation differences)
- Auto-compact trigger threshold raised from `AutoCompactThreshold()` to `0.975` of context window — simpler and more predictable
- `compactLocallyForSendBudget` now only does group truncation (no microcompact step)
- All microcompact-only tests removed; 4 dead constants and 2 helper functions cleaned up

### Multi-file tools: auto-merge duplicate paths

The `multi_file_read`, `multi_file_edit`, and `multi_file_write` tools previously rejected duplicate paths with an error, which was overly strict — the LLM frequently groups edits logically but accidentally repeats a path.

- **`multi_file_read`**: Duplicate paths silently skipped (first occurrence wins)
- **`multi_file_edit`**: Duplicate paths auto-merged — all edits from both entries are concatenated and applied as one batch (with max-edits validation)
- **`multi_file_write`**: Duplicate paths use last-write-wins semantics

## Features

### GGCODE.md init prompt at startup

When opening a project that has no `GGCODE.md`, `AGENTS.md`, `CLAUDE.md`, or `COPILOT.md`, the TUI now shows a prompt asking whether to create one. Press `y` to initialize or `n`/`Esc` to skip.

The old startup banner ("Initializing...") has been removed entirely — it was always hidden and served no purpose.
