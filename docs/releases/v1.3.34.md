## v1.3.34

### Tunnel Broker Projection Sync

The broker now serializes authoritative snapshot seeding with live event delivery. When a mobile client connects and the broker publishes a snapshot (or replays canonical events), all subsequent `Push*` calls block until the projection sync completes. This prevents snapshot and live events from interleaving on the client side, which previously caused missing messages or duplicated tool results on mobile.

- **`projectionMu` / `projectionCond`** — condition-variable barrier that gates all `Push*` methods during snapshot seeding
- **`clientReplayInFlight`** — coalesces duplicate client-connect triggers into a single authoritative replay pass
- **`clientProjectionSeeded`** — atomic flag that distinguishes the first client join (needs full snapshot) from additional joins (reuses relay retained history)
- **`sendSnapshotDirect` / `seedHistoryDirect`** — extracted from `SendSnapshot` to bypass the outbound queue and guarantee ordering within the projection sync window

### Tunnel Session Resume Revalidation

When a desktop session is resumed and re-shared to mobile, the broker now revalidates the persisted canonical tunnel ledger. If the ledger only covers pre-restart messages but the session has new post-restart content, the stale ledger is downgraded (`TunnelEventsComplete = false`, events cleared) so the broker falls back to the current chat snapshot instead of replaying outdated history.

### Tunnel Connected Notice Deduplication

The "mobile connected" system message is now shown exactly once per share session. Receiving multiple client-connect events (e.g., from relay reconnection) no longer produces duplicate notices in the chat area or pollutes tunnel event history.

### Tunnel History: Empty Tool Results

Completed tool calls (success, error, canceled) now always emit a `tool_result` entry in tunnel history, even when the result body is empty. Previously, tools that returned no text were invisible on mobile because the `tool_result` was skipped when the body was blank.

### Flutter Connect Coalescing

Concurrent `connect()` calls to the same URL are now coalesced — the second call awaits the first instead of creating a duplicate connection. This prevents race conditions when the UI triggers multiple connection attempts in quick succession.

### Flutter Projection Authority Guards

Three new guards prevent stale cached projections from overriding authoritative relay data:

1. **`full_history` resume** — When the relay sends `resume_ack` with `resume_mode: full_history`, the client skips local cache restoration entirely since the relay will stream the complete history.
2. **`snapshot_reset` handling** — After receiving a `snapshot_reset`, the client marks the projection as authoritative, preventing late-arriving cache restore callbacks from reseeding stale data.
3. **`active_session` race** — When an `active_session` event arrives shortly before a `full_history` resume, the earlier async cache-restore task is suppressed to avoid cursor conflicts.
