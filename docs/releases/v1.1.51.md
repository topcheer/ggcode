# ggcode v1.1.51

## Highlights

- **WebUI with real-time chat** — built-in web interface for chatting with the agent from any browser, works in both TUI and daemon modes.
- **Interactive ask_user buttons** — `ask_user` choice questions now render as clickable buttons on Telegram, Discord, Feishu, and Slack instead of plain text.
- **Five IM daemon fixes** — ask_user premature timeout, `/restart debug` swallowed by pending ask, duplicate fallback text, context cancellation, and TOCTOU race.

## New Features

### WebUI Chat

ggcode now starts a built-in HTTP server (127.0.0.1, random port) with:

- **WebSocket chat** — send messages and receive real-time streaming responses from the agent.
- **Image/file attachments** — attach images or files to messages for multi-modal input.
- **Session browser** — browse past session history grouped by workspace.
- **Config + MCP management** — REST API for configuration and MCP server management.
- **SPA frontend** — embedded single-page application served from `dist/` or fallback `index.html`.

In TUI mode, the WebUI URL is displayed as a system message inside the chat area. In daemon mode, it works alongside the IM gateway.

Architecture: `ChatBridge` interface decouples WebUI from the agent — `TUIChatBridge` routes messages through `program.Send()` into the Bubble Tea event loop; `DaemonBridge` injects them through `pendingInterruptions` into the agent loop. No direct agent access from WebUI prevents concurrency issues.

### Interactive ask_user Buttons

When the agent calls `ask_user` with choice questions, adapters that support `InteractiveSender` now render clickable button cards instead of plain text:

| Platform | Button Support |
|----------|---------------|
| Telegram | ✅ InlineKeyboard buttons |
| Discord | ✅ Button components |
| Feishu | ✅ Card action buttons |
| Slack | ✅ Block Kit buttons |
| QQ | ❌ Plain text fallback |
| DingDing | ❌ Plain text fallback |

Fallback logic: text is sent **only** to adapters that don't implement `InteractiveSender` — no duplicate messages on any platform. Unified in `IMEmitter.EmitAskUserInteractive()` for both TUI and daemon modes.

### `/restart debug`

Running `/restart debug` from any IM channel now enables `GGCODE_DEBUG=1` for the next daemon launch, making it easy to toggle debug logging remotely.

## Fixes

### IM Daemon ask_user Fixes (5 issues)

1. **Premature timeout** — `SubmitInboundMessage` used the SDK event callback's `ctx` for `context.WithCancel()`. When the SDK cancelled the event context (WS reconnect, event processing complete), the agent run and `ask_user` were cancelled prematurely. Fixed by using `context.Background()` instead.

2. **`/restart debug` swallowed** — In `SubmitInboundMessage`, the `pendingAsk` check came before slash command handling. When ask_user was pending, all text messages (including `/restart debug`) were consumed as ask_user responses. Fixed by moving slash command handling first.

3. **Duplicate fallback text** — `ask_user` interactive buttons were sent to adapters with `InteractiveSender`, then fallback text was sent to **all** adapters including those that already received buttons. Fixed by using `EmitToNonInteractive()` which sends fallback only to adapters that don't implement `InteractiveSender` (QQ, DingDing).

4. **Context cancellation propagation** — agent run context was bound to SDK event context, causing premature cancellation on WS reconnects.

5. **TOCTOU race in SendUserMessage** — idle-start path had a time-of-check-to-time-of-use race. Fixed with single mutex lock covering both check and start.

### Feishu Card Callback

- Feishu button callbacks now work via SDK's `OnP2CardActionTrigger` handler (upgraded SDK to v3.6.1).
- Single WS connection handles both message events and card callbacks, avoiding the error where two WS connections for the same appID cause disconnection.
- Card buttons include correct `behavior: "callback"` in button definition.

### WebUI

- Agent history loads correctly in WebChat on connect.
- Concurrent requests rejected with 429 status.
- Image/file attachments validated and processed correctly.
- WebSocket uses per-connection write goroutines (buffered channel of 256) to prevent concurrent read/write.
