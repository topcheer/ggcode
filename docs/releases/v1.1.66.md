# ggcode v1.1.66

## Highlights

Wires harness auto-run into all three entry points (TUI, pipe, daemon), implements strict mode semantics, fixes AutoInit minimal config, and corrects harness skill metadata.

## Changes

### ShouldAutoRun wired into TUI/pipe/daemon
- **TUI** (`internal/tui/commands.go`): Before starting the normal agent, `checkAutoRun()` evaluates the input. RouteSuggest shows the suggestion message; RouteHarness delegates to `/harness run`.
- **Pipe** (`cmd/ggcode/pipe.go`): Before running the agent, `checkPipeAutoRun()` evaluates the input. RouteHarness delegates to harness (currently prints guidance; inline task creation is Phase 4).
- **Daemon** (`internal/im/daemon_bridge.go`): Before each agent run, `checkAutoRunSuggestion()` logs harness routing suggestions. In daemon mode this is informational — the agent uses skill instructions to decide whether to invoke harness.

### Strict mode semantics
- `AutoRunResult.StrictWriteGuard` is set when `auto_run: "strict"` and a project is available.
- In strict mode, `worktree_mode` is overridden to `"required"` — PrepareWorkspace will fail if a git worktree cannot be created (no auto-fallback to repo root).
- TUI's `applyStrictWriteGuard()` provides the hook point for permission policy enforcement.

### AutoInit minimal config fix
- `MinimalAutoInitConfig()` produces a harness config with NO required files, NO content rules, NO check commands. This ensures auto-initialized projects pass their own checks out of the box.
- New test `TestAutoInit_MinimalConfigPassesChecks` verifies this.

### Harness skill metadata fix
- All four harness skills (`harness-run`, `harness-review`, `harness-promote`, `harness-diagnose`) are now `UserInvocable: false` — model-visible only, not slash commands. They are triggered by the auto-run router or agent skill matching.

## Test Coverage

- `auto_run_test.go`: +2 tests (strict mode, on mode) = 11 total
- `router_test.go`: 26 tests (unchanged)
- All existing tests pass, `make verify-ci` green.
