# v1.3.86

## Desktop: Session Loading + LAN Chat Rendering

### Bug Fixes

**Desktop session auto-load** — Desktop now iterates all workspace sessions
(newest-first) and loads the first one with actual messages that isn't locked
by another instance, mirroring TUI behavior. Previously it only tried the
single newest session and fell through to creating an empty one if that was
locked.

**Session lock release on close** — `Close()` now explicitly releases the
session lock so other instances can load it on restart. Previously, desktop
exit left zombie locks that prevented other instances from loading sessions.

**Empty session skip** — Auto-load now skips sessions with zero messages,
preventing stale auto-created empty sessions from taking priority over real
conversations.

**Compaction history preservation** — Desktop now sets a checkpoint handler
so compaction appends a checkpoint record to the JSONL file instead of
letting `saveSession()` do a full rewrite that would destroy all
pre-compaction history. The `saveSession()` method also detects compaction
(agent has fewer messages than session) and skips the full rewrite.

**Agent context restoration on session load** — `InitAgent` now calls
`RestoreSessionIntoAgent` to populate the agent context with the loaded
session's messages. Without this, the agent started with an empty context
and the first `saveSession()` call would overwrite the session JSONL with
just 1-2 messages.

**Frontend session loading race** — ChatView now self-fetches the session ID
from the backend as a fallback when the parent Layout's mount polling hasn't
completed yet. This fixes the blank screen on startup where the session was
loaded in Go but the frontend never received the session ID.

**Frontend new/switch session state reset** — When sessionId changes, old
messages are now cleared immediately. Previously, switching to an empty new
session left stale messages from the previous session visible.

### Improvements

**LAN Chat message rendering** — Agent-to-agent messages injected via LAN
Chat are now rendered with:
- A small "LAN Chat from xxx" label above the message (similar to IM/Mobile
  source badges)
- Markdown rendering for message content
- Stripped injection prefix ("New user guidance arrived...") from display
- Parsed `[LAN Chat from nick]:` prefix to extract the sender label

**LAN Chat agent receipt suppression** — Receipts for agent-to-agent messages
are no longer shown in the chat panel. Only human-to-human message delivery
status is visible.

**LAN Chat agent message persistence** — Agent-to-agent messages injected
via `SendHiddenText` now properly save the session and flush tunnel state,
matching the `SendMessage` code path.

**TUI LAN Chat rendering** — Agent-to-agent messages in TUI now render as
markdown user messages (via `chatWriteUserMarkdown`) instead of plain text.
History session loading also uses `chatWriteUserMarkdown` for user messages.

**TUI broadcast is now team-scoped** — The `/chat` broadcast command and the
`lanchat` `broadcast` action now send to the user's own team members only,
falling back to global broadcast if no team members are online.

### lanchat Tool Changes

**New `broadcast` semantics**:
- `broadcast` — sends to **your own team** (team-scoped, respects isolation)
- `broadcast_all` (new) — sends to **everyone on the LAN** (was old `broadcast`)
- `send` with `to='*'` — equivalent to `broadcast_all`

**Multi-recipient DM** — `send` now accepts comma-separated node IDs:
```
lanchat(action='send', to='id1,id2,id3', message='…')
```

**Default `to_role`** — When `as_agent=true`, `to_role` defaults to `agent`
(was `human`), which is the more common use case for agent-to-agent messaging.
